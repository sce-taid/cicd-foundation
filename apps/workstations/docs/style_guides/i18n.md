# Internationalization (i18n) Style Guide

This guide defines the standards for internationalization and localization across all components of the repository.

## 1. Universal Mandates

- **Full Coverage:** All user-facing text MUST be internationalized. No hardcoded strings are permitted in source code.
- **UN Language Support:** All features MUST support the 6 official United Nations languages: Arabic (ar), Chinese (zh), English (en), French (fr), Russian (ru), and Spanish (es).
- **Placeholder Consistency:** Go template placeholders (e.g., `{{.Field}}`) or language-specific equivalents MUST be preserved exactly across all translations.

## 2. Localization Files (JSON)

- **Standardized Sorting:** All localization JSON files (e.g., `locales/*.json`) MUST be kept alphabetically sorted by their `id` field. This ensures deterministic diffs, prevents duplicate keys, and facilitates side-by-side comparison between languages.
- **Key Naming:** Use `snake_case` for translation IDs (e.g., `error_file_not_found`).
- **Formatting:** Use 2-space indentation for JSON files.

## 3. Technical Error Strings

To maintain professional and readable logs:

- **Case:** Technical error strings (IDs starting with `error_` or `failed_`) SHOULD be lowercase.
- **Punctuation:** Technical error strings MUST NOT end with punctuation (e.g., use "failed to load file" instead of "Failed to load file.").
